Runner

by: Grace Gabrielson

This game was made for my level 2 project at The League of Amazing Programmers.
Click here to download and play!

1. Why I Made This Game

I decided to make this game to understand how infinite games work. I was inspired by Temple Run and wanted to create a game with the same concept. It is exciting to play a popular game, like Temple Run, and be able to replicate it, even in simpler terms.

2. What I Learned

User testing is vital in game development. The welcome screen to my game was more complicated than necessary, so my brother thought I should have the three difficulty buttons transfer directly to the game instead of using the "enter" key to start. He also thought that the player should die if the square goes off the screen or off the path. I made both of these adjustments. I also asked my other brother for his input, and he thought that there was not enough variation in the difficulties. To fix this, I increased the over all speed in the medium and hard difficulties. I decreased the spawn time of the obstacles to make them appear faster in the harder difficulties. A friend thought it would be cool to keep track of the player's high score, so in the "end of the game" screen, I included the user's current score, as well as their record score.

3. Architecture of My Game

My game utilizes nine different classes. The RunnerGame class sets up the window and adds the Game State object. The GameState class is where the different screens are created, updated, and drawn. My coin, obstacle, and player class extend the RunnerObject class to use the super constructor for dimensions, location, and the collision box. The RunnerManager uses an object of the RunnerSquare class and array lists of the class types Coin and Obstacle. The RunnerManager update the RunnerSquare, Coins, and Obstacles; the update methods are called in the GameState class using the The RunnerManager object "manager." The RunnerManager contains methods that checks collisions of the different game elements, purges objects, and increases the game difficulty.

4. Technical Implementation

I am proud of how I was able to get the path to move towards the player. The path that the square was playing on is a trapezoid with horizontal, moving lines. The lines help to make it seem as though the square is moving forward. Because the path is larger at the bottom I had to use the slope of the lines to increase the size of the lines as they moved down. Once I had the lines moving realistically I needed to change the movement of the obstacles and coins. Moving straight down did not look realistic so I used a variable to determine how far away the objects were from the center. That value was used to vary the angle that the objects would move down the screen at.

5. Most Challenging Part

The most challenging part of this project was making sure that everything was coming at the player at the same speed. I used a base speed to have consistency with the speed of the lines, gaps, obstacles, and coins. The gaps created a problem with the obstacles because obstacles would appear over the gap. To fix this problem, I implemented a boolean to determine if an obstacle could be spawned. Whenever I added a hole, which was every twelve frames, I set the boolean to be false. It was set back to true using a timer for the gaps.

6. Future Fixes and Updates

To improve this game, I would add different themes or characters for the player to choose from. I could include turns, in contrast to only a straight path. It would be interesting to see if I could keep track of who was playing by name and keep track of their individual score. With this, I could implement a leaderboard, so multiple people would be able to see their own scores and compare them.
